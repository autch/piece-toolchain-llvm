# SRF33 ファイルフォーマット仕様書

> **出典**: S5U1C33000C Manual (C Compiler Package for S1C33 Family) Ver.4, Appendix pp.475–480
>
> **エラッタ**: piece-lab 2015年の検証により `e_scnndx` フィールドのサイズ誤記が判明（後述）

---

## 概要

SRF33 (Seiko Epson Relocatable File Format for S1C33) は、EPSON S1C33ファミリ用ツールチェーン（as33アセンブラ、lk33リンカ）が使用するオブジェクトファイルおよびライブラリファイルのフォーマットである。

本文書ではオブジェクトファイル（A-1）とライブラリファイル（A-2）の両方の構造を記述する。

> **注意**: lk33の`-ld`コマンドで作成したローダ用srf33ファイルは、通常のsrf33ファイルとは構造が異なる。ローダ用srf33については `utility/ld33/` ディレクトリ内の readme を参照。

---

## 用語

- **チェーン**: ファイル先頭からのバイト数で、続く情報のつながりを示す。0の場合は続く情報はない。
- **インデックス**: セクションやシンボルを識別するための番号で、先頭が0となる。

---

## A-1 srf33 オブジェクトファイルの構造

### ファイル全体レイアウト

```
┌─────────────────────────┐
│ srf33制御ヘッダ情報      │
├─────────────────────────┤
│ セクション情報 1          │ ─→ リロケーション情報 1
│                         │ ─→ エクスターン情報 1
│                         │ ─→ 実データ 1
├─────────────────────────┤
│ セクション情報 2          │ ─→ リロケーション情報 2
│                         │ ─→ エクスターン情報 2
│                         │ ─→ 実データ 2
├─────────────────────────┤
│ ...                     │
├─────────────────────────┤
│ セクション情報 n          │ ─→ リロケーション情報 n
│                         │ ─→ エクスターン情報 n
│                         │ ─→ 実データ n
├─────────────────────────┤
│ デバッグ制御情報 1        │ ─→ ファイル名情報 1
│                         │ ─→ 文情報 1
│                         │ ─→ シンボル情報 1
├─────────────────────────┤
│ デバッグ制御情報 2        │ ─→ ファイル名情報 2
│                         │ ─→ 文情報 2
│                         │ ─→ シンボル情報 2
├─────────────────────────┤
│ ...                     │
├─────────────────────────┤
│ デバッグ制御情報 m        │ ─→ ファイル名情報 m
│                         │ ─→ 文情報 m
│                         │ ─→ シンボル情報 m
└─────────────────────────┘
```

ファイル内の配置順序は以下のとおり（フラット表現）:

1. srf33制御ヘッダ情報
2. セクション情報 1〜n
3. リロケーション情報 1〜n
4. エクスターン情報 1〜n
5. 実データ 1〜n
6. デバッグ制御情報 1〜m
7. ファイル名情報 1〜m
8. 文情報 1〜m
9. シンボル情報 1〜m

---

### 各情報の概説

#### セクション情報
- リロケータブルモジュールの場合: CODE, DATA, BSS 各1個ずつ計3個
- アブソリュートモジュールの場合: 1〜3個（CODE, DATA, BSS それぞれ0個または1個）
- リンク後のモジュールでは、リロケータブルな連続セクションはまとめられるが、アドレスを個別に指定したセクションおよびアブソリュートセクションはそのまま独立して存在する
- リンク後のセクション情報は CODE→DATA→BSS の順にまとめられ、BSSセクションもセクション情報が出力される
- 各セクション情報の内容: セクション属性（CODE/DATA/BSS）、配置アドレス、リロケーション情報/エクスターン情報/実データへのチェーン、実データのサイズなど。また次のセクション情報へのチェーンも持つ

#### リロケーション情報
- リンカの再配置に必要な情報で、アセンブラが生成するオブジェクトファイルに含まれる
- リンク後のオブジェクトファイルには出力されない
- 内容: セクション内の命令コードに対するリロケーションタイプ、セクション内の位置、参照するエクスターン情報へのインデックスなど

#### エクスターン情報
- リンクに必要なシンボル情報で、アセンブラが生成するオブジェクトファイルに含まれる
- リンク後のオブジェクトファイルには出力されない
- 内容: セクション内で使用されているシンボルの名称と種類、セクション内の位置など

#### 実データ
- 各セクションごとの実際のデータ
  - **CODEセクション**: データなし（BSSセクションは常に0）
  - **DATAセクション**: 1つのデータに1, 2, または4バイトを使用
- CODEセクションの場合、1つのコード（命令）に2バイト（下位→上位の順）で出力される

#### デバッグ制御情報
- リンクしたオブジェクトモジュール数だけ作られる
- それぞれのモジュールのファイル名情報、文情報、シンボル情報のサイズと、チェーンへのポインタを持つ
- モジュールをリンクした順序で並ぶ
- デバッグ制御情報およびファイル名情報・文情報・シンボル情報は、Cコンパイラgcc33またはプリプロセッサpp33からリンカlk33まで`-g`オプションを指定して実行させた場合に出力される
- デバッグ制御情報を持たない、あるいは読み出しに失敗した場合でも、実データまでは正しく読み込まれれば、逆アセンブル表示によるデバッグは行える

#### ファイル名情報
- `.file`擬似命令で生成されるファイル名情報
- 各モジュールのファイル名およびインクルードしたファイルのディレクトリ構造の情報を持つ
- 主にデバッガがソースコードを表示する際に参照

#### 文情報
- `.loc`および`.endfile`擬似命令で生成される行番号情報、ファイル切り換え情報
- 主にデバッガが実データとソースコードを対応させるために参照

#### シンボル情報
- モジュール内で定義されているすべてのシンボルの情報を持つ
- シンボルの表示およびシンボルを使用したアドレス指定の際に参照

### (4) エクスターン情報

> **⚠️ エラッタ**: マニュアル（p.478）では `e_scnndx` が **4バイト** と記載されているが、実際には **2バイト** が正しい。piece-lab 2015年の実バイナリ検証により確認済み。他のセクションIDフィールド（`s_scnndx`, `r_scnndx`）もすべて2バイトであり、整合する。

### (5) 実データ

- **CODEセクション**: 1つのコード（命令）が2バイト（下位→上位の順、つまりリトルエンディアン）で出力される
- **DATAセクション**: 1つのデータが1バイト（byte）、2バイト（half word）、または4バイト（word）で出力される。バイトオーダーは下位→上位（リトルエンディアン）
- **BSSセクション**: 実データは持たない

| y_namsiz  | シンボル名の長さ            | 1    | `0x00`〜`0x20`                                                |
| y_symnam  | シンボル名                 | *    | シンボル名, 最大32バイト                                        |

**合計サイズ: 15 + シンボル名長（可変）+ AUXエントリ**

#### <AUXエントリ>

`y_auxcnt` の数だけ、シンボル情報の直後に続く。

| フィールド | 情報                      | Byte | 内容                                                          |
|-----------|--------------------------|------|---------------------------------------------------------------|
| x_tagndx  | 構造体,共用体,enumタグ名へのインデックス | 4 | `0x00000000`〜`0xffffffff`                                  |
| x_size    | 構造体,共用体,配列のサイズ   | 4    | `0x00000000`〜`0xffffffff`                                   |
| x_dimen   | 配列のディメンション        | 4*4  | `0x00000000`〜`0xffffffff` × 4個（`0xffffffff`×4の計16バイト）  |

**AUXエントリサイズ: 24バイト（固定）**

> 注: `x_dimen` は4次元まで対応し、各次元が4バイトで合計16バイト。

## A-2 ライブラリファイルの構造

> **⚠️ 重要**: 以下に記載するマニュアルのフォーマット（ライブラリヘッダ / オブジェクトモジュール情報 / ライブラリシンボルテーブル）は、**実際の `.lib` ファイルのバイナリ構造と一致しない**。
>
> P/ECE SDK の実ファイル（lib.lib, idiv.lib, fp.lib 等）を実バイナリ解析した結果、実際は `!<lib33>` マジックで始まる全く異なるフォーマットが使われていることが判明した（`srf2elf` 実装時に確認）。
>
> **実際のフォーマット仕様は `docs/lib33_format.md` を参照すること。**
>
> 以下はマニュアルの記載を原文のまま保存しているが、実装には使用しないこと。

---

### ライブラリファイル全体レイアウト（マニュアル記載、実バイナリと不一致）

```
┌─────────────────────────┐
│ ライブラリヘッダ          │
├─────────────────────────┤
│ ライブラリシンボルテーブル 1│
│ ...                     │
│ ライブラリシンボルテーブル n│
├─────────────────────────┤
│ オブジェクトモジュール情報 1│
│   └→ オブジェクトファイル 1 │
├─────────────────────────┤
│ ...                     │
├─────────────────────────┤
│ オブジェクトモジュール情報 n│
│   └→ オブジェクトファイル n │
└─────────────────────────┘
```

#### ライブラリヘッダ
ライブラリファイルは必ずライブラリヘッダを1つ、ファイルの先頭に持つ。ライブラリヘッダはファイルサイズやライブラリファイル名の情報、最初のオブジェクトモジュールへのポインタ（ファイル内の位置情報）で構成される。

#### オブジェクトモジュール情報
オブジェクトモジュール個々に、オブジェクトモジュール情報が生成される。オブジェクトモジュール情報はオブジェクトファイル名やサイズ、対応するライブラリシンボルテーブルへのポインタで構成される。

#### ライブラリシンボルテーブル
個々のオブジェクトモジュールに対応する、グローバルシンボルテーブルである。グローバルシンボルを持たないオブジェクトモジュールには生成されない。ライブラリシンボルテーブルは、シンボルテーブルのサイズ、個々のグローバルシンボル情報（シンボル名、対応するオブジェクトモジュール情報へのポインタ）で構成される。

---

## エラッタまとめ

### エラッタ 1: e_scnndx サイズ誤記（マニュアル p.478）

| 項目 | マニュアル記載 | 実際の値 |
|------|-------------|---------|
| `e_scnndx` のサイズ | 4バイト | **2バイト** |

**発見者**: piece-lab / Naoyuki Sawa (2015-09-29)

**検証方法**: サンプルアセンブリソースを `pcc33 -c` でコンパイルし、生成されたオブジェクトファイルのバイナリダンプを手動で解析。`e_scnndx` を4バイトと解釈するとフィールド境界が合わず、2バイトと解釈すると正しく整合することを確認。

**根拠**: 他のすべてのセクションIDフィールド（`s_scnndx`, `r_scnndx`, `t_scnndx`等）は2バイトであり、`e_scnndx` のみ4バイトであるのは不自然。

---

### エラッタ 2: バイトオーダー誤記（本文書の誤り）

| 項目 | 本文書初版の記載 | 実際の値 |
|------|---------------|---------|
| マルチバイトフィールドのバイトオーダー | リトルエンディアン | **ビッグエンディアン** |

**発見**: `srf2elf` 実装時 (2026年)

**検証方法**: `hello.srf` (P/ECE SDK サンプル) のバイナリを直接解析:
- `c_ver` の期待値 0x3300 がファイル上で `33 00` (上位バイト先) で格納 → BE確定
- `c_scnptr = 0x0010` がファイル上で `00 00 00 10` → BE確定
- セクション情報が44バイト間隔で連続することもBE解釈で確認

**補足**: S1C33コア自体はリトルエンディアン。ただし as33/lk33 は EPSON の他ツールと同様にオブジェクトファイルをビッグエンディアンで出力している。

---

### エラッタ 3: セクション情報サイズ誤記（マニュアルおよび本文書の誤り）

| 項目 | マニュアル記載 | 実際の値 |
|------|-------------|---------|
| セクション情報の合計サイズ | 42バイト | **44バイト** |

**発見**: `srf2elf` 実装時 (2026年)

**検証方法**: フィールドを直接積算: 4+2+2+2+4+4+4+4+4+4+4+4+2 = **44バイト**。また `s_nxptr` チェーンが44バイト間隔で連続することをバイナリで確認。`s_scnndx` の正しいオフセットは42（マニュアルが示す40ではない）。

## バイトオーダー

SRF33ファイル内のすべてのマルチバイトフィールドは **ビッグエンディアン**（上位バイトが先）で格納される。

> **⚠️ エラッタ**: 本文書の初版（および元マニュアル）では「リトルエンディアン」と記載していたが、誤り。実バイナリの検証により判明した（`srf2elf` 実装時に確認）:
>
> - `c_ver = 0x3300` → ファイル上のバイト列 `33 00`（BE）
> - `c_scnptr = 0x0010` → ファイル上のバイト列 `00 00 00 10`（BE）
> - セクション情報の s_nxptr が 44 バイト間隔で連続することを BE 解釈で確認
>
> S1C33コア自体はリトルエンディアンだが、as33/lk33ツールが生成するオブジェクトファイル形式はビッグエンディアンで設計されている。

## srf2elf 実装上の注意

1. **バイトオーダー**: SRF33の全マルチバイトフィールドはビッグエンディアン。ELF出力はリトルエンディアン（S1C33 ELFターゲット規約）。パーサは `struct.unpack_from('>...', ...)` を使用すること
2. **セクション情報サイズ**: 44バイト（マニュアル記載の42バイトは誤り）。`s_scnndx` のオフセットは 42
3. **e_scnndx エラッタ**: エクスターン情報の `e_scnndx` は必ず2バイトとしてパースすること
4. **チェーンの解釈**: チェーン値 = 0 は「続く情報なし」を意味する。チェーン値が非ゼロの場合はファイル先頭からのバイトオフセット
5. **可変長フィールド**: シンボル名やファイル名は `*_namsiz` フィールドで長さが先行する。パディングはない
6. **ライブラリファイルの処理**: 実際の `.lib` ファイルはA-2に記載のフォーマットと異なる。`!<lib33>` マジックで始まる別形式。詳細は `docs/lib33_format.md` を参照
7. **デバッグ情報**: srf2elfでは実データとリロケーション/エクスターン情報が重要。デバッグ制御情報（ファイル名・文・シンボル情報）は `-g` オプション付きでビルドされた場合のみ存在し、DWARF変換が必要な場合を除き初期実装ではスキップ可能
8. **リロケーションタイプのマッピング**: `r_rctyp` の各値はS1C33の `ext` 命令によるアドレス分割パターンに対応する。ELFリロケーションタイプへの変換が必要
